# Literary Studio — техническое задание

Приложение для написания книг. Платформы: десктоп (основная рабочая среда) + Android (просмотр, лёгкая правка). Синхронизация между устройствами.

---

## 1. Структура данных

```
Серия
 ├─ Библия мира — НЕ отдельный файл, а агрегированный вид из коллекций ниже
 │   ├─ Персонажи (scope: серия/локальный; ядро + состояния по книгам)
 │   ├─ Локации (scope: серия/локальный)
 │   ├─ Организации (scope: серия/локальный)
 │   ├─ Правила (карточка: название + формулировка + исключения)
 │   └─ Термины (глоссарий: термин + определение)
 ├─ Авторский стиль (серия-уровень)
 │   ├─ Стоп-лист оборотов
 │   ├─ Пунктуационные правила (внутренняя речь, диалоги)
 │   ├─ POV / время повествования
 │   └─ Тональность (краткое описание для AI-контекста)
 └─ Книга
     ├─ Реквизиты: заголовок, подзаголовок, жанр, теги, аннотация (краткая/полная) — создание книги открывает пустую карточку реквизитов (без мастера-визарда), с табами Реквизиты/Продакшн/Публикация в одном экране
     ├─ Главы
     │   └─ Сцены (текст)
     ├─ Идеи (список с тегами для группировки — не доска/канбан; черновик-накопитель, не путать с Библией мира)
     ├─ Матрица романа (сюжетные линии × биты)
     ├─ Таймлайн (хронология событий внутри мира, к нему привязаны состояния персонажей по книгам)
     ├─ Карта связей (mindmap персонажей/организаций/локаций)
     ├─ Продакшн-ассеты (обложки, промпты трейлеров, TTS, мерч)
     ├─ Версии (история правок, diff, откат)
     └─ Корзина (удалённые главы/сцены/персонажи — восстановимо)
```

**Важное разграничение:** Идеи — черновик «а что если», Библия мира — устоявшийся канон, который не противоречит сам себе. Персонажи/Локации/Организации/Правила/Термины — не части одного файла, а собственные коллекции карточек; Библия мира — сформированный из них индекс на уровне серии, не отдельное хранилище. Авторский стиль (включая Стоп-лист) — тоже на уровне серии.

---

## 2. Режимы работы

Разделение на два режима с разным набором инструментов в UI:

### Планирование
- Матрица романа
- Таймлайн
- Карта связей
- Карточки арок персонажей
- Библия мира
- Идеи

Ячейки матрицы и таймлайна не исчезают при переходе к написанию — они остаются живым чек-листом.

### Написание
- Редактор глав/сцен
- Статистика (счётчики, темп, плотность)
- AI-помощники применительно к тексту

Переключение между режимами — не разные приложения, а разные наборы панелей вокруг одной и той же книги.

## 3. Сущности планирования

### 3.1 Матрица романа
Таблица: строки — сюжетные биты (номер + краткое событие), столбцы — сюжетные линии (не обязательно персонажи — могут быть организации, локации, мета-линия «общий поворот»).

- Ячейка матрицы **линкуется на сцену**, когда та написана — статус ячейки автоматически отражает «сцена написана / не написана»
- Столбцы редактируемые (добавить/убрать линию)
- Экспорт матрицы в CSV/таблицу для печати

### 3.2 Таймлайн
Хронология событий **внутри мира** (даты/возраст персонажей/прошедшее время между событиями) — отдельно от порядка повествования в матрице. Нужен для проверки континуити (не разошлось ли время между сценами).

- Точки на шкале, привязка к сценам
- Возможность нелинейного порядка повествования при линейном таймлайне (флэшбеки и т.п.)

### 3.3 Карта связей (mindmap)
Узлы — персонажи/организации/локации, рёбра — подписанные связи («работает на», «враг», «скрывает от»).

- Drag-перенос узлов
- **Добавление узла на граф** — через кнопку, открывающую выбор существующей сущности из коллекции (Персонажи/Локации/Организации), а не создание новой прямо на графе — создание новой сущности остаётся в соответствующем разделе Библии мира (§3.5), граф только расставляет то, что уже есть
- Автогенерация матрицы персонажей из графа связей (кто с кем связан → таблица)

### 3.4 Карточка персонажа + арка
Создаётся кнопкой «+ Новый персонаж» на самом экране списка Персонажей — открывает пустую карточку (тот же паттерн, что у создания книги, §1: пустая форма, не мастер из шагов).

Помимо базовых полей (имя, описание, роль):
- **Варианты имени + правило контекста** (например: Алекс — при обращении к нему, Эл — в неформальной речи, Грэм — фамилия в официальном контексте) — используется AI-проверкой на согласованность
- **Таблица арки**: строки — ключевые сцены персонажа, столбцы — Цель / Внутренний конфликт / Что узнал / Отношение к [другому персонажу]; отметка типа бита (завязка/испытание/перелом/развязка) на строку

### 3.5 Библия мира
Не отдельный документ, а **агрегированный вид** из структурированных коллекций серии — Персонажи, Локации, Организации, Правила, Термины. Каждая коллекция — карточки, а не проза, чтобы не расходиться с самими сущностями (иначе повторяется проблема «текст в двух местах», как раньше было с обложками/стоп-листом).

**Форма создания одинакова для всех четырёх типов** (Локация/Организация/Правило/Термин) — одна форма целиком, не мастер по шагам, с почти идентичным набором полей: Название · Категория/Тип (свободный тег) · содержательное поле (Описание для Локации/Организации, Формулировка+Исключения для Правила, Определение+Синонимы для Термина) · Область действия (серия/локальная) · Связанные сущности (чипы, добавляются по кнопке) · Источник/Первое упоминание (сцена) · Статус (черновик/утверждено).

- **Организации** — отдельная категория от Локации/Персонажа (в матрице романа WildMind — организация, не место)
- **Правила** — карточка: Название (короткая метка) · Категория (физика мира/технология/социальное устройство/лингвистика, свободный тег) · Формулировка (конкретно, не проза) · Исключения/ограничения (отдельное поле) · Область действия (серия/локальная) · Связанные сущности (персонажи/локации/организации, линк через карту связей §3.3) · Источник — сцена первого объяснения читателю · Статус (черновик/утверждено)
- **Термины** — глоссарий: Термин · Определение · Синонимы/альтернативные названия · Категория (технология/организация/понятие/сленг) · Область действия · Первое упоминание — сцена, где термин впервые вводится · Связанные сущности

Поле «Источник/Первое упоминание» у обеих карточек решает конкретную проблему многокнижной серии: не дать термину или правилу появиться в тексте раньше, чем читатель до него дошёл по сюжету — AI-проверка может сверять порядок использования с порядком сцен.

**Область действия (scope):** у каждой карточки — `Серия` (канон, виден и редактируется из любой книги серии) или `Локальный` (только эта книга, не засоряет общую библию разовыми персонажами).

**Состояния по книгам:** канон персонажа/локации/организации не меняется, но статус может — жив/погиб, переехал, изменил цели. Это не переписывание карточки, а список состояний, привязанных к Таймлайну (§3.2) конкретной книги — так библия не противоречит сама себе между книгами серии.

AI-помощники используют весь агрегированный вид как контекст автоматически при работе с текстом любой книги серии — снижает риск противоречий канону.

### 3.6 Авторский стиль
Отдельная сущность на уровне серии (не книги), подключается к AI-помощникам так же автоматически, как Библия мира:
- **Стоп-лист оборотов** — переносится между книгами серии, используется при финальной вычитке (см. §5). Список пар «штамп → пометка для переформулировки» с примерами: *«сердце ёкнуло» → переформулировать без штампа*, *«глаза расширились» → описать реакцию иначе*. Добавление оборота — вручную кнопкой, либо прямо из текста сцены (выделил фразу → «Добавить в стоп-лист»)
- Пунктуационные правила (например, оформление внутренней речи персонажей, диалогов)
- POV / время повествования, если зафиксированы
- Тональность повествования — краткое описание для AI-контекста, не инструкция каждый раз вручную

## 4. Написание

### 4.1 Редактор
- Дерево Главы → Сцены
- Serif-шрифт для текста, стандартное форматирование (жирный/курсив/цитата)
- Отменить/повторить, найти-и-заменить
- Целевой объём на сцену/главу/книгу (знаки), прогресс-бар

### 4.2 Статус написания (стадии, не просто чат с AI)
Формализованный пайплайн вместо произвольного общения с помощником:

`Черновик → На редактуре → Вычитано → Финал`

Статус на уровне сцены/главы, переключается выпадающим списком в тулбаре редактора (не отдельным экраном) — виден и меняется без ухода со страницы.

**Что даёт статус:** матрица романа (§3.1) показывает не только «написано / не написано» одним индикатором, а разные состояния готовности — ячейка сцены в статусе «Черновик» и ячейка в статусе «Финал» визуально различимы (например, интенсивность заливки или отдельная точка-индикатор рядом с галочкой). Так на общей картине книги сразу видно не просто, где есть текст, а где текст уже готов, а где ещё сырой — иначе статус был бы полезен только внутри одной сцены и не влиял на общую картину книги.

### 4.3 Статистика
**Живёт в разделе Планирование** (не только в редакторе) — сводный экран книги, объединяющий базовые счётчики и график темпа в одном месте, а не разнесённый по разным экранам:

- Базовое: слов, знаков, знаков без пробелов — на сцену/главу/книгу (в редакторе — сокращённая строка для текущей сцены, §12; здесь — сводно по всей книге)
- Прогресс: цель в знаках, выполнено сегодня, streak (дни подряд с прогрессом)

### 4.4 Темп и плотность текста
Часть того же экрана «Статистика» (не отдельный пункт меню). Два независимых, накладываемых на один график слоя:

**Автоматический (без AI, чистая статистика):**
- Средняя длина предложения по сцене
- Доля диалога vs описания (эвристика по кавычкам/тире)
- Соотношение «объём текста / длительность события по таймлайну» (сцена на 3000 знаков про 5-минутный разговор ощущается медленнее той же длины про месяц пути)

**Субъективный (ручная оценка или AI-подсказка от помощника-«аналитика темпа»):**
- Оценка напряжения сцены 1–10, отложенная по оси сцен книги — кривая напряжения по всей книге (аналог кривой формы истории)

Обе кривые — на одном графике под книгой, проседания видны сразу.

**Расположение:** проверка темпа — периодическая (раз в несколько глав, §4.4 выше), не после каждой сцены, поэтому график не встроен постоянно в статистику редактора — это добавило бы визуальный груз ради того, что смотрится редко. График живёт на едином экране **«Статистика»** в разделе Планирование, вместе со сводными счётчиками книги (§4.3) — не как два похожих отдельных пункта меню, а один экран для периодического обзора.

## 5. AI-помощники

- 4 системных помощника, применимых к любому полю/тексту — формализованный состав по опыту редакторского пайплайна серии:
  1. **Со-автор** — генерация и продолжение текста, черновые варианты. Действия: «Продолжить», «Переписать», «Предложи вариант»
  2. **Критик** — структурный анализ (дыры в сюжете, нестыковки, соответствие Библии мира); источник оценки напряжения сцены для графика темпа (§4.4)
  3. **Редактор** — построчная правка: грамматика, стиль, проверка по Стоп-листу и правилам пунктуации из Авторского стиля (§3.6), проверка согласованности имён персонажей (§3.4)
  4. **Читатель** — реакция «первого взгляда»: понятность, эмоциональный отклик, что автор перестаёт замечать из-за замыленности глаза
- Возможность задавать своих (кастомные персоны с системным промптом)
- Контекст помощника при работе с текстом книги автоматически включает Библию мира и Авторский стиль серии (включая Стоп-лист, §3.6) — без ручного копирования контекста в каждый запрос
- Отдельный помощник/режим — «аналитик темпа», подсказывающий оценку напряжения сцены (см. §4.4)
- Проверка согласованности имён персонажей по правилам из карточки (§3.4) — по всему тексту книги
- Проверка на совпадения из Стоп-листа (§3.6 Авторский стиль) при финальной вычитке — подсветка найденных оборотов (одновременно инлайн-подчёркиванием в тексте редактора и списком результатов в боковой панели — оба варианта, не один вместо другого)

### 5.1 Формат ответа помощников — структурированный отчёт, не просто чат
Ответ помощника не свободный текст в чате, а структурированные пункты с прямой ссылкой на место в тексте — иначе замечание нельзя ни отследить, ни быстро применить. Пример для **Критика**:

**Системный промпт (основа, дополняется контекстом Библии мира/Авторского стиля/Таймлайна автоматически):**
```
Ты — литературный критик, разбирающий структуру сцены/главы.
Не редактируй стиль и грамматику — это задача Редактора.
Отвечай только в формате списка находок. Для каждой находки укажи:
- Тип: [дыра в сюжете / нестыковка с Библией мира / провисание темпа / нарушение таймлайна]
- Место: цитата-якорь (5–10 слов из текста, не длиннее)
- Суть: что не так, в одном-двух предложениях
- Серьёзность: [критично / стоит проверить / на усмотрение автора]
Если находок нет — так и скажи, не выдумывай замечания.
```

**Пример ответа в UI:**
```
⚠ Критично · Нестыковка с Библией мира
Место: «...Майкл впервые увидел лабораторию...»
Суть: по карточке Локации «Лаборатория» Майкл уже бывал там в Сцене 2 —
      здесь описано как первое посещение.

◐ Стоит проверить · Провисание темпа
Место: «...прошло ещё некоторое время...»
Суть: третий абзац подряд без диалога и действия — сверить с графиком
      темпа (§4.4), возможно стоит сократить.
```
Каждая находка кликабельна — переход к месту в тексте по цитате-якорю. Тот же принцип (системный промпт + структурированный список находок с местом/сутью/серьёзностью) применяется к Редактору и Читателю, с адаптацией типов находок под их роль.

## 6. Версии и Корзина

- **Версии**: два источника создания снимков —
  - *Периодическое автосохранение* — раз в 5–10 минут по умолчанию, интервал настраивается в Настройках
  - *Событийное сохранение* — при переходе между сценами/пунктами меню (сохраняем то, что редактировали, перед уходом), создании главы/сцены/персонажа, смене статуса (§4.2), закрытии приложения
- **Diff между версиями** — визуальное сравнение, а не слепой откат. Без diff непонятно, что теряешь при откате
- **Откат — два режима навигации к нужной версии**:
  - *По шагам* — N версий назад
  - *По времени* — состояние текста на определённый момент (точка на временной шкале, а не порядковый номер), когда не помнишь количество правок, но помнишь примерно когда всё было в порядке

  Оба режима ведут к одному и тому же diff-просмотру перед подтверждением отката
- **Корзина**: удалённые главы/сцены/персонажи хранятся восстановимо, не удаляются насовсем сразу

**Визуал экрана версий:** двухпанельный — слева список снимков, сгруппированный по дням, с иконкой источника (часы — автосохранение, флаг — смена статуса, дверь — закрытие приложения) и меткой времени; справа — сравнение выбранной версии с текущей: убранный текст зачёркнут на приглушённо-красном фоне, добавленный — на приглушённо-зелёном, тем же принципом, что и diff в обычных редакторах кода. Кнопки «Откат по шагам» / «Откат по времени» переключают способ навигации по левому списку, не меняя сам просмотр diff. Подтверждение отката — отдельная кнопка снизу, не по клику на версию в списке (защита от случайного отката).

## 7. Импорт/экспорт

- Импорт текста (для переноса существующих черновиков) — пункт меню **Файл** (§12, верхнее меню десктопа)
- Экспорт: **FB2, DOCX, EPUB — приоритетные форматы для публикации на платформах**. PDF — только для превью/печати, не для загрузки на площадки: на опыте Litres загрузка PDF блокирует генерацию аудиокниги и требует 14 дней снятия с публикации для исправления. Вызывается тем же пунктом меню **Файл**, не отдельной кнопкой в интерфейсе книги — выбор формата в подменю
- Экспорт матрицы романа отдельным файлом (таблица)

## 8. Продакшн-ассеты (за пределами текста)

Раздел на уровне книги — таб «Продакшн» на том же экране, где Реквизиты (§1), версионируется вместе с текстом:
- Обложки (с версионированием — на опыте были итерации типа cover_v5)
- Промпты для трейлеров/видео (например Kling AI)
- TTS-файлы (аудио)
- Файлы под мерч

**Визуал:** обложка с превью-миниатюрой, подписью текущей версии и ссылкой на историю версий; промпты трейлера — карточки с источником (например «Kling AI · сцена 1») и текстом; TTS и мерч — компактные списки файлов с иконкой типа.

## 9. Издательский трекинг

Лёгкий раздел «Публикация» на книгу — таб на том же экране, где Реквизиты и Продакшн (§1, §8):
- Платформа, статус (черновик / отправлено / опубликовано / снято с публикации)
- SEO-поля площадки (если применимо)
- Заметки по площадочным ограничениям (например: только fb2/docx/epub с самого начала, не PDF)

**Визуал:** карточка платформы (иконка + название) со статусным бейджем справа (та же цветовая система статусов, §13 UI kit); поля SEO — обычные текстовые поля; предупреждение по ограничениям площадки — жёлтая плашка (тот же принцип, что тосты-предупреждения, §13), не мелкий серый текст, который легко пропустить. Кнопка «+ Добавить платформу» — книга может публиковаться на нескольких площадках одновременно.

## 11. Синхронизация

Десктоп — основная рабочая среда (полный набор инструментов планирования и написания). Android — просмотр и лёгкая правка текста, статистика, статусы.

**Модель: сервер-БД как источник правды + офлайн-кэш на клиенте с очередью синхронизации** (как Google Docs/Notion), а не симметричное локально-первое хранение и не ручной экспорт/импорт как основной канал.

Выбор продиктован тем, что текущая версия задумана как многопользовательский сайт — сервер должен быть авторитетным источником данных, а не одной из равноправных копий (в отличие от чисто личного офлайн-инструмента, где симметричная локально-первая модель была бы уместнее).

- БД на хостинге — источник правды, там же живёт многопользовательская логика
- Клиент (десктоп/Android) кэширует нужные данные локально; работа офлайн (например, в самолёте) не требует заранее выполненных действий — правки складываются в локальную очередь
- При появлении сети очередь синхронизируется с сервером
- Конфликт (правки на двух устройствах офлайн одновременно) не перезаписывается молча — разрешается через diff-интерфейс, уже заложенный в §6 (Версии)
- Ручной экспорт (FB2/DOCX/EPUB, архив проекта) остаётся — как функция бэкапа/переноса/публикации, не как основной механизм синхронизации между сессиями
- Если позже продукт свернётся до личного инструмента без многопользовательского сайта — модель данных не потребует переделки, изменится только обязательность постоянного сервера (вопрос деплоя, не архитектуры данных)

## 12. Визуальное описание для реализации

### Общие принципы
- Плоский минималистичный UI, много воздуха, без градиентов и теней (кроме функциональных — фокус-ринг)
- Serif-шрифт только в зоне текста сцены; весь остальной UI — sans-serif
- Различать иконками два типа взаимодействия в списках: `chevron-right` — переход на другой экран, `chevron-down` — разворачивание на месте (см. мобильные мокапы, переданные ранее)
- Деструктивные действия (удаление) никогда не доступны одним тапом/кликом без промежуточного подтверждения (bottom sheet на мобильном, контекстное выпадающее меню на десктопе — появляется у точки клика по «⋮», список действий с разделителем перед удалением, которое подсвечено красным; это меню, не модальное окно на весь экран — на десктопе для простого набора действий модалка избыточна)
- **Настройки на десктопе** — та же панель, что описана для мобильного (§12 мобильный раздел: Профиль, Файл, Вид, Справка, Выйти), открывается по клику на шестерёнку в шапке как выпадающая панель у правого края экрана, не отдельный полноэкранный переход

### Десктоп — общая раскладка
Самый верх — тонкая строка классического меню приложения: Файл, Редактировать, Вид, Руководство, О программе (десктопу, в отличие от мобильной версии, хватает места, чтобы не прятать это в настройки — на мобильном те же пункты собраны в меню настроек, см. мобильный раздел ниже).

Под ней — хлебные крошки Серия › Книга › Глава › Сцена (та же логика, что в мобильной шапке — только горизонтально разворачивается, без обрезания текста), справа — переключатель Планирование/Написание, профиль, настройки.

Трёхколоночная рабочая область:
- **Левая колонка** (дерево навигации, ~150–280px, сворачиваемая), сгруппирована по разделам:
  - Главы → Сцены (в режиме «Написание»)
  - Планирование: Матрица, Таймлайн, Карта связей, **Статистика** (сводка по книге + график темпа, §4.3–4.4) — доступны как быстрые ссылки независимо от текущего режима, поскольку ячейки матрицы/таймлайна остаются живым чек-листом и во время написания (§2)
  - Библия мира: Персонажи, Локации, Организации, Правила, Термины — вложенные пункты, не самостоятельные разделы верхнего уровня (отражает архитектуру агрегации, §3.5)
  - Серия: Авторский стиль, Идеи, Продакшн
  - Корзина — внизу списка, за разделителем, приглушённым цветом (не должна визуально конкурировать с рабочими разделами)
- **Центральная колонка**: рабочая область — редактор сцены в режиме «Написание» ИЛИ матрица/таймлайн/mindmap в режиме «Планирование» (переключатель — в верхней панели, не вкладками внутри рабочей области)
- **Правая колонка** — панель AI-помощника, **присутствует в обоих режимах** (в режиме «Планирование» полезна для анализа таблицы/графа — «посмотри на матрицу и скажи, где проседает линия Х»), сворачиваема в обоих. Ширина регулируется перетаскиванием (resize-хендл между редактором/рабочей областью и панелью), а не фиксирована — особенно важно в режиме «Написание», где ответы помощника (Со-автор и др.) могут быть длинными и нуждаются в ширине, сопоставимой с центральной колонкой

### Матрица романа (визуал)
Таблица на всю ширину рабочей области, строки-биты слева пронумерованы, столбцы — сюжетные линии с возможностью горизонтального скролла при большом числе линий. Ячейка со связанной сценой — акцентная заливка фона + небольшая галочка перед текстом (не рамка/полоса слева — при нескольких связанных ячейках подряд в строке рамки создают лишний узор поверх границ таблицы; заливка+иконка читается чище и не требует различения цвета для доступности). Клик открывает связанную сцену. Пустая ячейка (сцена не написана) — приглушённая, с возможностью создать и сразу связать сцену прямо из неё. Ячейка «—» (линия не участвует в этом бите) визуально отличается от «не написано», чтобы не путать два разных состояния. Кнопки над таблицей: добавить сюжетную линию, экспорт матрицы (§7).

### Таймлайн (визуал)
Горизонтальная шкала с точками-событиями, привязанными к датам/относительному времени. Точки соединены с соответствующими сценами (клик — переход). Плотность точек визуально показывает, где событий в мире много, а где — временной разрыв.

### Карта связей (визуал)
Граф force-directed или drag-layout: узлы — карточки-чипы с именем/иконкой роли, цвет чипа определяется категорией (персонаж/организация/локация — три фиксированных цвета с легендой в шапке рабочей области), рёбра — подписанные линии (например «работает на», «враг», «хранитель»). Клик по узлу подсвечивает его рёбра акцентным цветом среди остальных линий графа и открывает карточку узла в правой панели, не перекрывая граф.

Правая панель здесь — не отдельная четвёртая колонка поверх панели AI, а тот же слот с переключателем **Карточка / AI** сверху: по умолчанию показывает карточку выбранного узла, кнопка «Спросить AI про [узел]» в карточке переключает на вкладку AI с уже подставленным контекстом узла — не нужно объяснять помощнику, о чём речь. Тот же принцип применим к Матрице и Таймлайну: там, где появляется контекстная карточка/деталь, AI-панель не пропадает, а становится соседней вкладкой в том же слоте.

### Карточка персонажа с аркой (визуал)
Верх — базовые поля (имя, варианты имени, роль, описание). Ниже — таблица арки в виде горизонтальных карточек-этапов (завязка → испытание → перелом → развязка), каждая с короткими полями Цель/Конфликт/Что узнал, разворачиваемыми по клику.

### Статистика и прогресс (визуал)
Одна строка под редактором, сокращённые подписи (Слов / Зн / Без проб), не «Знаков»/«Без пробелов» целиком — экономит горизонтальное место. Цель показана и числом (выполнено/цель), и мини-прогресс-баром рядом — заполнение считывается взглядом без вычисления процента в уме.

### Статистика (визуал)
Единый экран в разделе Планирование, сверху вниз: три метрик-карточки (всего знаков, % от цели книги с мини-прогресс-баром, серия дней подряд), затем прогресс-бары по каждой главе (заполнение = % написанного), затем график темпа — линейный, по оси X сцены по порядку, две наложенные кривые разного цвета: автоматический темп (сплошная линия) и ручная оценка напряжения (пунктир). Легенда минимальная, без перегруза цветами.

### Мобильная версия (Android)
Иерархия левого дерева синхронизирована с десктопом (§ выше): Главы, Планирование (Матрица/Таймлайн/Карта связей/Статистика — с пометкой «просмотр», read-only на Android), Библия мира с вложенными Персонажи/Локации/Организации/Правила/Термины (не отдельные разделы верхнего уровня), Серия (Авторский стиль/Идеи/Продакшн), Корзина внизу приглушённым цветом.

Прочие паттерны, ранее закреплённые в мокапах (архив `literary-studio-mobile-mockups.zip`): разделение chevron-right/chevron-down, bottom sheet для действий и AI-инструментов, единая шапка с контекстом книги/главы/сцены, единая строка статистики без дублирования.

**AI на мобильном** — нет постоянной боковой панели, как на десктопе; вместо неё карточки объектов (персонаж, узел карты связей и т.п.) несут кнопку «Спросить AI про [объект]», открывающую bottom sheet AI с уже подставленным контекстом — тот же принцип, что и десктопная кнопка в панели «Карточка», просто другой механизм вызова, уместный для узкого экрана.

**Цветовая система категорий** (персонаж/организация/локация — фиолетовый/коралловый/тил) идентична десктопу — общий визуальный язык между платформами, не два независимых набора цветов.

**Арка персонажа на мобильном** — горизонтальный скролл карточек-этапов (не 2×2 сетка, как в среднем сценарии между экранами), т.к. на узком экране скролл читается естественнее сжатой сетки.

Планирование (матрица/таймлайн/mindmap) на Android — в режиме просмотра (read-only или ограниченная правка), полноценное редактирование этих модулей — прерогатива десктопа.

**Контекст шапки зависит от уровня сущности, не копируется бездумно со сцены на всё подряд**: экраны текста сцены (Редактор, Помощники) показывают «Книга / Глава / Сцена». Экраны сущностей Библии мира (персонаж, локация и т.п.) показывают контекст серии — «Серия · Библия мира › Персонажи» — потому что такая сущность обычно не привязана к одной сцене или даже одной книге (§3.5, scope серия/локальная).

**Единая строка статистики (слов/знаков/цель) — только на экранах текста сцены.** На карточках Библии мира, Планирования и прочих нетекстовых экранах она не показывается — это метрика написанного текста, а не общий элемент шаблона экрана.

**Нижний таб-бар (Коллекция/Редактор/Помощники) сохраняется на всех экранах внутри дерева библиотеки**, включая карточки персонажей/локаций — пользователь остаётся «внутри» раздела Коллекция, просто на несколько уровней вглубь, и таб должен оставаться подсвеченным активным, а не пропадать.

**Редактор на мобильном** несёт те же решения, что и десктопный (§4.2, §5): статус сцены — выпадающий список в тулбаре; подсветка стоп-листа — волнистое подчёркивание прямо в тексте, а не отдельная боковая панель (на мобильном для неё нет места) — найденное совпадение показывается инлайн-карточкой сразу под текстом с возможностью проигнорировать.

**Планирование на мобильном (read-only) — единый паттерн адаптации**: широкие десктопные виды (таблица Матрицы, граф Карты связей, график Статистики) на узком экране превращаются в вертикальный список карточек — один бит/узел/показатель на карточку, без попытки втиснуть таблицу или граф в 380px. Внизу экрана — подсказка «Полное редактирование — на десктопе». Этот паттерн одинаков для Матрицы, Таймлайна, Карты связей и Статистики — отрисован один раз (Матрица) и переиспользуется остальными без содержательных отличий.

**Формы на мобильном** (Реквизиты книги, карточки Библии мира, Авторский стиль) — тот же набор полей, что на десктопе, но в одну колонку на весь экран вместо табов/двух колонок — обычная адаптация формы под узкий экран, без изменения состава полей.

**Экран «Помощники» на мобильном обновлён** под структурированный формат ответа (§5.1): горизонтальный скролл переключателей между 4 системными помощниками сверху, ответ — не свободный текст, а карточки находок с типом/местом/серьёзностью, как и на десктопе.

**Версии на мобильном** — список снимков (без второй панели рядом, как на десктопе); тап по версии открывает diff на весь экран, с теми же цветами добавлено/убрано и кнопкой отката снизу.

### Системные экраны на мобильном
UI kit, авторизация/регистрация, тосты и индикатор сохранения — те же паттерны и цветовые правила, что на десктопе (§13), адаптированные в одну колонку на весь экран. Модалка действий (десктопный контекстный дропдаун, §12) на мобильном остаётся bottom sheet, как было решено изначально — не переносится один в один.

## 13. Системные экраны

Не относятся к работе с книгой напрямую, но без них нет рабочего приложения — до этого прохода не были спроектированы вовсе.

### Типовые элементы (UI kit)
Единый набор состояний для кнопок (первичная/вторичная/текстовая/опасная/недоступна), полей форм (по умолчанию/фокус/ошибка с текстом под полем), статусных бейджей (переиспользует цвета статусов сцены §4.2: Черновик/На редактуре/Вычитано/Финал + Ошибка), переключателей (чекбокс/радио/тумблер). Разрабатывается один раз и переиспользуется во всех формах (Реквизиты книги, карточки Библии мира, Авторский стиль) — не изобретается заново на каждом экране.

### Авторизация / регистрация / вход
Поскольку продукт задуман как многопользовательский сайт (§11), это не опциональный экран. Вход: email + пароль, ссылка «Забыли пароль?», альтернативный вход через Google, переход к регистрации. Регистрация — аналогичная форма с дополнительным полем подтверждения. Экран без верхнего меню/дерева навигации — отдельный, вне основной структуры приложения.

### Уведомления и ошибки
Два разных механизма, не взаимозаменяемые:
- **Тост-уведомления** (нижний угол экрана, исчезают сами) — для некритичных событий: подтверждение сохранения, предупреждение о разрыве сети (правки сохраняются локально), ошибка с кнопкой «Повторить» для восстановимых сбоев (например, не удался экспорт). **Цвет несёт статус, не только иконка**: успех — нейтральный фон (не окрашивать каждый раз, когда всё в порядке), предупреждение — жёлтый фон, ошибка — красный фон; текст и иконка используют тот же цвет, что и фон, а не нейтральный поверх цветной плашки
- **Модальное окно** — только для ситуаций, требующих решения пользователя, не автоматически проходит само. Пример — конфликт версий при синхронизации (§11): правка на другом устройстве, нужно выбрать «Сравнить» перед продолжением, простым тостом не обойтись, потому что нельзя дать конфликту потеряться незамеченным

### Индикатор сохранения
Нижний правый угол экрана, всегда на виду вне зависимости от того, какой экран открыт:
- **Зелёная точка** — сохранено недавно
- **Жёлтая точка** — давно не сохранялось (правки есть, но давно не улетали на сервер)
- **Красная точка** — офлайн, изменения в локальной очереди синхронизации (§11), не потеряны, просто ещё не отправлены

Наведение/тап показывает точную метку времени последнего сохранения («12 мин назад»). Красное состояние — это про офлайн-очередь, не про «давно», поэтому у него отдельный цвет, а не просто более тревожный оттенок жёлтого.
